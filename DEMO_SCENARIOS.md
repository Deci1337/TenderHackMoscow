# Сценарии демонстрации для защиты

Для 5-минутной презентации. Один разработчик говорит, второй показывает в браузере.

---

## Сценарий 1: Исправление опечаток и морфология (25 баллов — семантика)

**Цель:** Показать NLP-пайплайн — pymorphy2 + pg_trgm.

**Шаги:**
1. Войти как «Новый пользователь» (кнопка в правом нижнем углу).
2. Ввести запрос с опечаткой: `ноутбуки`  
   *Ожидаемо:* Найдены результаты. Над выдачей: «Показаны результаты по запросу "ноутбук"».
3. Ввести запрос с опечаткой: `принтир`  
   *Ожидаемо:* pg_trgm исправляет на «принтер» через fuzzy matching.
4. Ввести форму в косвенном падеже: `школьных парт`  
   *Ожидаемо:* Лемматизатор приводит к «школьный парта», tsvector находит «Парта школьная».

**Что говорим:** «Система использует pymorphy2 для лемматизации — приводит все словоформы к базовой. Fuzzy matching через pg_trgm обрабатывает опечатки без словаря. Внешних API нет — всё локально.»

## Сценарий 2: Персонализация по истории (25 баллов — персонализация)

**Цель:** Показать, что выдача разная для разных пользователей на один запрос.

**Шаги:**
1. Переключить на «Школа №1234» (Образование).
2. Ввести: `бумага`  
   *Ожидаемо:* В топе — «Бумага офисная А4», «Бумага для принтера». Бейдж: «Популярно в вашей сфере: Образование».
3. Переключить на «Городская больница №5» (Здравоохранение).
4. Ввести: `бумага`  
   *Ожидаемо:* В топе — «Бумага медицинская», «Бумага для ЭКГ». Выдача изменилась.

**Что говорим:** «Одно слово "бумага" — два разных результата. Система знает профиль организации и её историю контрактов. Это обработка омографов через контекст пользователя.»

## Сценарий 3: Динамическая индексация + негативные сигналы (25 баллов)

**Цель:** Показать, что система учится прямо в рамках сессии.

**Шаги:**
1. Войти как «Школа №1234».
2. Ввести: `ручка`  
   *Запомнить порядок выдачи (Позиция 1, 2, 3...).*
3. Нажать «Подробнее» на карточке — и **сразу** вернуться на страницу поиска (< 3 сек — bounce).  
   *На фронтенде автоматически отправился `bounce` event.*
4. Снова ввести: `ручка`  
   *Ожидаемо:* Появился синий баннер: «Выдача изменилась, потому что в этой сессии вы отклонили 1 товар(ов)». Товар, от которого отказались, опустился ниже.
5. Нажать кнопку «Сравнить» на другой карточке.
6. Ещё раз ввести: `ручка`  
   *Ожидаемо:* Карточка со сравнением поднялась выше.

**Что говорим:** «Без перезагрузки индекса. Каждое действие пользователя — клик, bounce, сравнение — немедленно (через Redis, <1 мс) влияет на следующий поиск. Это и есть динамическая индексация.»

## Сценарий 4: Объяснимость ранжирования

**Цель:** Показать прозрачность системы.

**Шаги:**
1. Открыть любую карточку СТЕ с бейджами.
2. Указать на конкретный бейдж, например «Вы уже закупали этот товар (3 раза)».
3. Указать на другой товар с бейджем «Снижено: вы отклонили похожий товар».

**Что говорим:** «Каждая позиция в выдаче объяснена. Пользователь видит не "чёрный ящик", а конкретную причину. Жюри и заказчик могут в любой момент проверить логику ранжирования.»

## Ключевые фразы для ответов на вопросы жюри

**«Почему не использовали готовые поисковые движки?»**  
> Ограничения хакатона запрещают внешние API. Мы построили поиск на PostgreSQL с расширениями pgvector и pg_trgm — это легковесное self-hosted решение с полным контролем над ранжированием.

**«Почему не GPT/Claude для обработки запросов?»**  
> Запрещено правилами. Мы используем pymorphy2 (~5 МБ) для морфологии и rubert-tiny2 (~118 МБ) для семантики — это открытые модели с быстрым CPU-инференсом, соразмерные нашим ресурсам.

**«Как масштабируется решение?»**  
> Смотри `SCALING.md`.
